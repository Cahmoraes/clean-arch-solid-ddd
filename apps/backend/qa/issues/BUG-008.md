# BUG-008: `Entity.id` getter retornava `string | undefined`, exigindo `entity.id!` em 11+ call sites

**Severity:** Medium
**Priority:** P2
**Type:** Data
**Status:** Resolved

## Environment

- **Build:** main @ 2dd985d (anterior à bateria de correção 2026-05-03)
- **OS:** N/A
- **Browser:** N/A
- **URL:** N/A (problema de domínio compartilhado)

## Summary

O Value Object `Id` aceitava ausência de identificador na construção via `Id.create()`, expondo `value: string | undefined`. Como consequência, todas as entidades (`User`, `Gym`, `CheckIn`, etc.) tinham getter `id: string | undefined`, forçando o uso de `entity.id!` em mais de 11 call sites — um anti-pattern que tanto bloqueava o lint (`noNonNullAssertion`) quanto criava risco real de runtime caso uma entidade fosse manipulada antes de receber id.

## Reproduction

```bash
# Antes:
grep -rn "\.id!" apps/backend/src | wc -l
# 11+ ocorrências em controllers, providers e factories
pnpm --filter backend biome:fix
# erro lint/style/noNonNullAssertion
```

Observed before the fix:

- 11+ usos de non-null assertion (`!`).
- Lint bloqueado.
- Risco de `TypeError: Cannot read properties of undefined` em manipulação parcial.

## Expected

- `Entity.id: string` sempre definido.
- Zero non-null assertions em call sites.
- `Id.create()` sem argumento gera UUID v4 automaticamente.

## Root cause

`Id.create()` foi modelado para "talvez não ter id" (ex.: nova entidade antes do persist). Esse "talvez" vazou pelo getter público em vez de ser confinado dentro do construtor da entidade. Consequência: tipo nullable propagado para toda a aplicação.

## Fix

- `apps/backend/src/shared/domain/value-object/id.ts` — `Id.create()` sem argumento agora chama `randomUUID()`. Getter `value: string` (não-nullable).
- `User`, `Gym`, `CheckIn` e demais entidades — getter `id: string` (não-nullable).
- Removidos os `?? randomUUID()` e `?? undefined` mortos em repositórios in-memory e Prisma.

## Verification

- `pnpm --filter backend biome:fix` → 0 erros.
- `pnpm --filter backend tsc:check` → 0 erros (eliminou os 32 erros TS prévios em conjunto com a refatoração R5).
- `pnpm --filter backend test:run` → 292/292.

## Impact

- **Users Affected:** Indireto — risco de bug em qualquer fluxo que tocasse `entity.id` antes do persist.
- **Frequency:** Latente (não disparado em produção, mas presente em todo call site).
- **Workaround:** `entity.id!` (justamente o anti-pattern eliminado).
