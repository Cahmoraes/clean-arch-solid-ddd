# BUG-001: CreateCustomerController event-driven sem teste de integração

**Severity:** High
**Priority:** P1
**Type:** Functional
**Status:** Resolved

## Environment

- **Build:** main @ 2dd985d (resolvido em 2026-05-03 — 2ª bateria de QA)
- **OS:** N/A (Node.js backend)
- **Browser:** N/A
- **URL:** Fluxo interno — evento `userCreated` → `CreateCustomerController`

## Summary

O `CreateCustomerController` é inicializado via IoC e subscreve ao evento de domínio `userCreated`. Quando um usuário é criado, este controller deve invocar o `CreateCustomerUseCase` para provisionar o customer no Stripe. Porém, não existia nenhum teste que validasse esta integração event-driven. Se o listener não estivesse registrado corretamente ou a subscrição falhasse silenciosamente, o usuário não teria `customerId` no Stripe e `POST /subscriptions` falharia com `BillingCustomerNotProvisionedError`.

## Reproduction

```bash
# Antes do fix:
find apps/backend/src -name "*create-customer*business-flow*"   # vazio
find apps/backend/src -name "*create-customer*integration*"     # vazio
```

Observed before the fix:

- Nenhum teste cobria o caminho `POST /users` → publish `userCreated` → CreateCustomer → Stripe gateway.

## Expected

Existir um teste de integração que:
1. Inicializa o container IoC com `CreateCustomerController` resolvido pelo bootstrap.
2. Executa `POST /users` via HTTP (usando `serverBuildForTest`).
3. Verifica que o `TestingSubscriptionGateway` recebeu a chamada `createCustomer`.
4. Verifica que o usuário persistido tem `billingCustomerId` igual ao `customer.id` retornado.

## Root cause

O `CreateCustomerController` não é um controller HTTP — ele registra um listener de evento de domínio no método `init()`. Foi omitido dos business-flow tests por não ter endpoint HTTP, mas o fluxo de negócio depende criticamente dele.

## Fix

Criado `src/subscription/infra/controller/create-customer.controller.business-flow-test.ts`:

- Dispara `POST /users` via supertest contra `serverBuildForTest()` (que carrega o `subscription-module` e instancia o `CreateCustomerController`).
- Aguarda assincronicamente (`waitForCustomer`) o listener processar o evento.
- Asserta que `subscriptionGateway.customerByEmail(input.email)` retorna o customer com `email`, `name` e `metadata: { userId }` corretos.
- Asserta que o `User` persistido tem `billingCustomerId` igual ao `customer.id`.
- Inclui spy `vi.spyOn(subscriptionGateway, "createCustomer")` para garantir invocação única por evento.
