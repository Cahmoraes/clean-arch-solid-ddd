# BUG-004: Erros de validação retornavam mensagens AJV cruas em vez de Zod amigáveis

**Severity:** High
**Priority:** P1
**Type:** Functional
**Status:** Resolved

## Environment

- **Build:** main @ 2dd985d (anterior à bateria de correção 2026-05-03)
- **OS:** N/A
- **Browser:** N/A
- **URL:** Todos os endpoints com validação de body/params/query

## Summary

Quando um cliente enviava um payload inválido, o backend respondia com mensagens cruas no formato AJV (ex.: `body/email must match format "email"`) em vez do formato Zod amigável já formatado pelos controllers (`{ message, issues: [{ path, message }] }`). Isso quebrava o contrato com o frontend, que esperava o formato Zod, e expunha detalhes internos do validador.

## Reproduction

```bash
curl -X POST http://localhost:3333/users \
  -H "Content-Type: application/json" \
  -d '{"email":"invalid","password":"123"}'
# Antes:
# {
#   "statusCode": 400,
#   "error": "Bad Request",
#   "message": "body/email must match format \"email\""
# }
```

Observed before the fix:

- Mensagens em inglês, formato AJV.
- Frontend não conseguia mapear `path → field` para destacar campos.

## Expected

```json
{
  "message": "Validation error",
  "issues": [
    { "path": "email", "message": "Invalid email" },
    { "path": "password", "message": "String must contain at least 6 character(s)" }
  ]
}
```

## Root cause

O Fastify rodava o validador AJV padrão antes do controller; o `safeParse` do Zod presente em cada controller só executava se o AJV passasse. Para erros de schema, o caminho Zod nunca era atingido.

## Fix

- `apps/backend/src/shared/infra/server/fastify-adapter.ts` — implementação de `makeValidatorCompiler` Zod nativo via `setValidatorCompiler`, substituindo o AJV nas rotas com schema Zod.
- `apps/backend/src/shared/infra/openapi/openapi-schema-builder.ts` — schemas continuam compatíveis com OpenAPI mas validação efetiva passa pelo Zod.
- `pickZodSchema` extraído com type guard `isZodValidationKey` para reduzir complexidade cognitiva.

## Verification

- `pnpm --filter backend test:business-flow` → 44/44, incluindo assertions de payload de erro.
- Curl manual com payload inválido retorna formato Zod.

## Impact

- **Users Affected:** Todos os clientes da API (web, mobile, integrações).
- **Frequency:** Sempre que há erro de validação.
- **Workaround:** Frontend tinha branch ad-hoc para parsear formato AJV.
