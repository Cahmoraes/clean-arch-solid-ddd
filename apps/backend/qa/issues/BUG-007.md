# BUG-007: GET /gyms (search) retornava `latitude`/`longitude` flat em vez de `coordinate` aninhado

**Severity:** Medium
**Priority:** P2
**Type:** Functional
**Status:** Resolved

## Environment

- **Build:** main @ 2dd985d (anterior à bateria de correção 2026-05-03)
- **OS:** N/A
- **Browser:** N/A
- **URL:** `GET /gyms?search=...`

## Summary

O DTO de resposta da busca de academias expunha `latitude` e `longitude` como campos flat no nível da gym, contrariando o desenho DDD em que `Coordinate` é um Value Object que deve aparecer aninhado em `coordinate: { latitude, longitude }`.

## Reproduction

```bash
curl "http://localhost:3333/gyms?search=academia" -H "Authorization: Bearer <jwt>"
# Antes:
# { "gyms": [ { "id":"...", "title":"...", "latitude": -23.5, "longitude": -46.6 } ] }
# Esperado:
# { "gyms": [ { "id":"...", "title":"...", "coordinate": { "latitude": -23.5, "longitude": -46.6 } } ] }
```

Observed before the fix:

- Campos `latitude`/`longitude` flat.
- Inconsistência com outros endpoints de gym (`fetch-nearby`, `create-gym`) que já retornam aninhado.

## Expected

Coordenada sempre encapsulada no objeto `coordinate`, refletindo o Value Object do domínio.

## Root cause

`SearchGymUseCase.toDTO()` expandia o `Coordinate` em campos planos. O controller propagava sem reaninhar. Inconsistência com o restante do bounded context, que sempre serializa `coordinate` como objeto.

## Fix

- `apps/backend/src/gym/application/use-case/search-gym.usecase.ts` — `toDTO` retorna `coordinate: { latitude, longitude }`.
- `apps/backend/src/gym/infra/controller/search-gym.controller.ts` — schema OpenAPI atualizado para refletir o objeto aninhado.

## Verification

- `pnpm --filter backend test:run -- -t "search-gym"` → passa.
- `pnpm --filter backend test:business-flow` → 44/44.
