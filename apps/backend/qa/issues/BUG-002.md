# BUG-002: Três controllers HTTP sem business-flow-test correspondente (e dois deles eram dead code)

**Severity:** High
**Priority:** P0
**Type:** Functional
**Status:** Resolved

## Environment

- **Build:** main @ 2dd985d (resolvido em 2026-05-03 — 2ª bateria de QA)
- **OS:** N/A
- **Browser:** N/A
- **URL:** `PATCH /users/activate`, `PATCH /users/:userId`, `GET /check-ins/metrics/:userId`

## Summary

Três controllers HTTP estavam registrados (em arquivo) mas sem `*.business-flow-test.ts` correspondente:

1. `activate-user.controller.ts` → `PATCH /users/activate`
2. `update-user-profile.controller.ts` → `PATCH /users/:userId`
3. `metrics.controller.ts` (check-in) → `GET /check-ins/metrics/:userId`

A criação dos testes revelou que **dois deles eram dead code** — o IoC não os instanciava: `UpdateUserProfileController` e `MetricsController` retornavam 404 em produção. Severidade re-classificada de Medium/P1 para **High/P0** porque duas features documentadas estavam quebradas silenciosamente.

## Reproduction

```bash
# Antes do fix:
find apps/backend/src -name "*activate-user*business-flow*"    # vazio
find apps/backend/src -name "*update-user-profile*business*"   # vazio
find apps/backend/src -name "*check-in-metrics*business*"      # vazio

# Após criar os testes, falhas com 404:
pnpm --filter backend test:business-flow -- update-user-profile
# → PATCH /users/:userId responde 404 (controller não registrado)
pnpm --filter backend test:business-flow -- check-in-metrics
# → GET /check-ins/metrics/:userId responde 404 (controller não registrado)
```

Observed before the fix:

- Rotas `PATCH /users/:userId` e `GET /check-ins/metrics/:userId` retornavam 404 a clientes em produção.
- `activate-user` retornava `body: string` cru em respostas de erro (inconsistência com convenção `{ message }`).

## Expected

Cada controller HTTP deve:
- Estar registrado no bootstrap e ter binding IoC correto.
- Ter `*.business-flow-test.ts` testando happy path, 401, 400 e 422/4xx.

## Root cause

1. **`UpdateUserProfileController` (dead code):**
   - `USER_TYPES.Controllers.UpdateUserProfile` estava bound a `UserProfileController` (errado).
   - `USER_TYPES.UseCases.UpdateUserProfile` estava bound a `UserProfileUseCase` (errado).
   - `setupUserModule` não resolvia `Controllers.UpdateUserProfile`.
2. **`MetricsController` / check-in (dead code):**
   - Symbol `CHECKIN_TYPES.Controllers.Metrics` não existia.
   - Sem `bind(...)` no `checkInModule`.
   - Sem `resolve(...)` em `setupCheckInModule`.
3. **`ActivateUserController` (formato de resposta inconsistente):** usava `ResponseFactory.BAD_REQUEST({ body: <string> })` em vez de `{ message: <string> }`.

A ausência dos testes mascarou as três regressões.

## Fix

**Código de produção:**
- `src/shared/infra/ioc/module/user/user-module.ts` — bind correto de `UpdateUserProfileController` e `UpdateUserProfileUseCase` com seus próprios símbolos.
- `src/bootstrap/setup-user-module.ts` — adicionado `resolve(USER_TYPES.Controllers.UpdateUserProfile)`.
- `src/shared/infra/ioc/module/service-identifier/checkin-types.ts` — adicionado símbolo `Controllers.Metrics`.
- `src/shared/infra/ioc/module/check-in/check-in-module.ts` — bind do `MetricsController`.
- `src/bootstrap/setup-check-in-module.ts` — adicionado `resolve(CHECKIN_TYPES.Controllers.Metrics)`.
- `src/user/infra/controller/activate-user.controller.ts` — `body:` → `message:` em `BAD_REQUEST` e `UNPROCESSABLE_ENTITY`.

**Cobertura nova:**
- `src/user/infra/controller/activate-user.business-flow-test.ts` (4 casos)
- `src/user/infra/controller/update-user-profile.business-flow-test.ts` (4 casos)
- `src/check-in/infra/controller/check-in-metrics.business-flow-test.ts` (3 casos)
