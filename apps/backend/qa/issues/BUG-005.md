# BUG-005: GET /users retornava body vazio (sem campos `users` e `pagination`)

**Severity:** High
**Priority:** P1
**Type:** Functional
**Status:** Resolved

## Environment

- **Build:** main @ 2dd985d (anterior à bateria de correção 2026-05-03)
- **OS:** N/A
- **Browser:** N/A
- **URL:** `GET /users`

## Summary

O endpoint `GET /users` (admin-only, paginado) respondia `200 OK` mas com body `{}` — os campos `users` e `pagination` eram silenciosamente removidos antes do envio.

## Reproduction

```bash
curl -H "Authorization: Bearer <admin-jwt>" http://localhost:3333/users?page=1
# Antes:
# {}
```

Observed before the fix:

- Body sempre vazio.
- Lista de usuários inutilizável pelo painel admin.
- Business-flow-test `fetch-users.business-flow-test.ts` falhava no `expect(body.users).toBeDefined()`.

## Expected

```json
{
  "users": [ { "id": "...", "email": "...", "role": "MEMBER" } ],
  "pagination": { "page": 1, "perPage": 20, "total": 42 }
}
```

## Root cause

O `openapi-schema-builder.ts` gerava schemas de response sem `additionalProperties: true`. O `fast-json-stringify` (serializador padrão do Fastify) usa o schema para serializar, e qualquer propriedade não declarada explicitamente no schema é descartada. Como o schema não havia sido atualizado para incluir os campos, o body inteiro saía vazio.

## Fix

`apps/backend/src/shared/infra/openapi/openapi-schema-builder.ts` — passou a setar `additionalProperties: true` por padrão na conversão Zod → JSON Schema para responses, garantindo que campos não-modelados explicitamente continuem sendo serializados. Schemas que precisam de strip explícito devem optar via `.strict()` no Zod.

## Verification

- `pnpm --filter backend test:business-flow -- -t "fetch-users"` → passa.
- Curl manual com JWT admin retorna estrutura completa.
