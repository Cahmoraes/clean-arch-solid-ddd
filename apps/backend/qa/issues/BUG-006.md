# BUG-006: PATCH /check-ins/validate retornava `checkInId` em vez de `validatedAt`

**Severity:** High
**Priority:** P1
**Type:** Functional
**Status:** Resolved

## Environment

- **Build:** main @ 2dd985d (anterior à bateria de correção 2026-05-03)
- **OS:** N/A
- **Browser:** N/A
- **URL:** `PATCH /check-ins/:checkInId/validate`

## Summary

O endpoint de validação de check-in respondia com o campo `{ checkInId }` quando o contrato OpenAPI e o use-case estabeleciam `{ validatedAt: ISO-8601 }`. O cliente que esperava o timestamp de validação recebia o id do recurso.

## Reproduction

```bash
curl -X PATCH http://localhost:3333/check-ins/<id>/validate \
  -H "Authorization: Bearer <admin-jwt>"
# Antes:
# { "checkInId": "abc-123" }
# Esperado:
# { "validatedAt": "2026-05-03T10:00:00.000Z" }
```

Observed before the fix:

- Campo errado no body.
- Frontend não conseguia exibir o horário de validação.

## Expected

Response `{ validatedAt: string (ISO-8601) }` conforme o schema OpenAPI e o retorno do `ValidateCheckInUseCase`.

## Root cause

`validate-check-in.controller.ts` montava o response a partir de `params.checkInId` em vez de propagar o `validatedAt` retornado pelo use-case. Erro humano de mapeamento; o use-case já retornava o campo correto.

## Fix

`apps/backend/src/check-in/infra/controller/validate-check-in.controller.ts` — handler passou a retornar `{ validatedAt: result.value.validatedAt.toISOString() }` alinhado ao contrato.
