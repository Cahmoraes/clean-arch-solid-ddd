# BUG-001: Promise rejection não tratada em onExceeded

**Severity:** High
**Priority:** P1
**Type:** Functional
**Status:** Fixed

## Environment

- **Build:** commit 04d69b2 (branch monorepo-migration)
- **OS:** Linux
- **URL:** Qualquer endpoint com rate limiting ativo

## Summary

No callback `onExceeded` do `RateLimitPlugin`, a chamada `queue.publish()` retorna uma Promise mas era tratada com try/catch síncrono. Em caso de falha na publicação (RabbitMQ indisponível), a Promise rejection não seria capturada, causando um `UnhandledPromiseRejection` que poderia crashar o processo Node.js.

## Reproduction

```typescript
// Código original (antes do fix):
try {
  queue.publish(Exchanges.RATE_LIMIT_EXCEEDED, event)
} catch {
  // Nunca captura — queue.publish() retorna Promise
}
```

Observado antes do fix:
- Promise rejection não capturada quando RabbitMQ está indisponível
- Potencial crash do processo Node.js em produção

## Expected

A falha na publicação do evento deve ser silenciosamente ignorada (fire-and-forget) sem afetar a resposta HTTP 429 ao cliente.

## Root cause

`queue.publish()` é async (retorna `Promise<void>`), mas o código usava `try/catch` síncrono que não captura rejeições de Promise. O `.catch()` precisa ser chamado diretamente na Promise retornada.

## Fix

```typescript
// Código corrigido:
queue.publish(Exchanges.RATE_LIMIT_EXCEEDED, event).catch(() => {})
```

Mock do teste também atualizado:
- `vi.fn()` → `vi.fn().mockResolvedValue(undefined)` (retorna Promise)
- Cenário de erro: `mockRejectedValue(new Error(...))` para testar o `.catch()`
